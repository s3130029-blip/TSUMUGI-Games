# CLAUDE.md — タマゴガチャ（仮）

このファイルは、本リポジトリで Claude Code が従う**行動規約**です。
作業の前に必ず本ファイルと `docs/REQUIREMENTS.md` を読み、両者に従ってください。

---

## 1. プロジェクト概要

- ブラウザで動く、子ども向けタマゴガチャ（カプセルトイ）ゲーム。
- タマゴをタップ → ガタガタ → パカッ → モノが生まれる、を中心とした**演出重視**の作品。
- 完全サーバーレス。ログイン不要。GitHub Pages で公開し、**iPad Safari での快適なプレイを最優先**する。
- 詳細仕様は `docs/REQUIREMENTS.md` を参照（こちらが仕様の正本）。

## 4. 技術スタック・制約

### 採用
- **TypeScript + Vite**（開発サーバー・ビルド）。
- **アプリ全体の画面・レイアウト**（画面遷移／図鑑グリッド／ボタン／モーダル）は**素のDOM + CSS（flexbox/grid）**を基本とする。SVGの座標系でアプリのレイアウトは組まない。
- **タマゴ本体＋ひび割れ**は**SVG**でパスを描く（卵の自然なカーブ、ひびのジグザグ線をきれいに表現するため）。
  - **動き（揺れ・スクワッシュ&ストレッチ・パカッと開く）はSVG内のグループ（`<g>`）に対する CSS `transform` で付ける**。これはDOM要素と同様にGPU合成され軽量。
  - `<filter>`（ぼかし・グロー等）や`d`属性そのもののアニメーションは**避ける**（iOS Safariで重くなりやすい）。ひびが広がる表現は`stroke-dasharray`/`stroke-dashoffset`、または数段階のSVGを差し替える方式を優先する。
- **キラキラ等のパーティクル**は CSS か Canvas の軽い方を選ぶ。**個数が多くなる（目安30〜40を超える）場合はCanvas**を優先する（DOM/SVG要素を多数並べると合成レイヤーが増えて重くなるため）。
- **Vitest** でユニットテスト。
- 読み上げ：**Web Speech API（SpeechSynthesis）**。
- 効果音：**Web Audio API の合成音**を基本（音声ファイルは使うなら軽量なものに限る）。
- 保存：**localStorage**。
- モノのアセット：**初期版は絵文字（Emoji）**。`assetUrl` へ差し替え可能な設計にしておく。

### 制約（守ること）
- **サーバー・外部通信・ログイン・課金・広告・トラッキングは入れない。**
- 重い依存・大きなバンドルを避ける。フレームワーク導入は要相談（理由を添えて提案）。
- GitHub Pages のサブパス公開で**リンク切れしない**こと（`base` 設定・`import.meta.env.BASE_URL` を考慮）。
- iOS Safari の制約を守る（§7参照）。

---

## 5. 決定性ルール（重要）

- ガチャの抽選など乱数を使う箇所は、**シード可能な擬似乱数（seedable PRNG）**を経由する。
  - `Math.random()` を抽選ロジックに直接使わない。
- 抽選・確率分布・境界値は**シードを固定して Vitest で再現テスト**できるようにする。
- ロジック（抽選・セーブ計算）と演出（アニメーション・音）は**分離**し、ロジックは純粋関数に寄せる。

---

## 6. 禁止事項（やらないこと）

- 仕様書・CLAUDE.md と矛盾する実装を、確認なしに進めること。
- ユーザーの承認なくフェーズを飛ばす・大規模リファクタを始めること。
- 抽選ロジックでの `Math.random()` 直接使用。
- 外部APIアクセス、解析タグ、フォント/画像のCDN必須化など、オフライン動作を壊す依存の追加。
- iOSで無効と分かっている機能（`navigator.vibrate` 等）に手応えを依存させること（手応えは音・光・動きで表現）。
- テストの削除・スキップによる「緑化」。落ちるテストは原因を直す。
- 個人情報の収集・保存。

---

## 7. iPad / Safari 固有の遵守事項

- アニメーションは `transform` / `opacity`（GPU合成）中心。レイアウト再計算を避ける。
- hover前提のUIにしない。タップ領域は大きく（子どもの指でも押しやすく）。
- `meta viewport` に `viewport-fit=cover`。ダブルタップズーム／ピンチで演出が乱れないように。
- セーフエリア（`env(safe-area-inset-*)`）を考慮。
- 音は**ユーザー操作（タップ）起点**で再生。`AudioContext` は最初のタップで `resume()`。
- 読み上げの音声リストは**非同期読み込み**（`voiceschanged` を待つ／リトライ）。利用不可ならテキスト表示にフォールバック。
- `prefers-reduced-motion` を尊重（演出控えめモード）。

## 8. 開発フロー

- **いきなり実装しない。** まず計画（構成・段取り・設計概要）を提示し、ユーザーの承認を得てから着手する。
- `docs/REQUIREMENTS.md`「3. 開発フェーズ」に沿って、**フェーズ1（MVP）から順番に**進める。
- 各フェーズ完了時に、**`npm run dev` で実際に遊べる状態**にする（ビルドが通り、型エラー0、テスト緑）。
- 1フェーズ内でも、まとまった単位で動作確認しながら進める。
- 大きな設計判断（フレームワーク採用、データ構造変更など）は、理由を添えて**提案→承認**の形を取る。

## 10. コーディング規約

- TypeScript は `strict` を有効化。`any` を避ける。
- マジックナンバー（確率・時間など）は**定数として一箇所に集約**し、調整しやすくする。
- ロジックとUI/演出を分離。副作用（DOM・localStorage・音）はUI層／専用モジュールに寄せる。
- 関数・型・モジュールの責務を小さく保つ。
- 重要な意図・仮決めはコメント（日本語）で残す。

---

## 11. テスト方針

- **Vitest 対象**：
  - 抽選（確率分布・シード固定での再現性・境界値）。
  - セーブ/ロード（破損データのフォールバック、NEW判定、ダブり集計、コンプリート率の導出）。
  - アイテムマスタ整合性（ID一意性・必須フィールド欠落の検出）。
- 演出・読み上げは実機（iPad）での目視確認。
- フェーズ完了時に**型エラー0・テスト全緑**を維持する。

---

## 12. コミット規約

- 1コミット＝1つの意味のある変更。メッセージは日本語。
- 例：`feat: タマゴのガタガタ→パカッ演出を実装`、`test: 抽選の確率分布テストを追加`、`fix: GitHub Pagesのサブパスでアセットが404になる問題を修正`。
- フェーズ完了時はその旨が分かるメッセージを残す。

---

## 13. デプロイ（GitHub Pages）

- Vite の `base` をリポジトリ名に合わせる（プロジェクトページの場合 `/<repo>/`）。
- GitHub Actions でビルド＆Pages公開のワークフローを用意する。
- 公開URLを iPad のSafariで開いて遊べることを最終確認とする。

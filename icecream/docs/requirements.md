# アイスクリーム屋さんゲーム 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名
アイスクリーム屋さんゲーム（仮称）

### 1.2 目的
子供が楽しく遊べる、アイスクリーム作成ゲームを開発する。
Windows PC で開発し、GitHub Pages で公開し、手持ちの iPad（Safari）でも快適に遊べることを目的とする。

### 1.3 想定ユーザー
- 主対象：iPad を使う子供
- 補助対象：開発・確認を行う大人

### 1.4 開発方針
- 静的 Web アプリとして実装する
- バックエンドは使わない
- 外部 API は使わない
- 必須ビルドツールは使わない
- GitHub Pages でそのまま動くことを優先する
- 後からアイスの種類や演出を増やしやすい設計にする

---

## 2. ゲーム概要

プレイヤーは、客の注文に応じてコーン付きのアイスクリームを作る。

画面には「注文見本」が表示され、プレイヤーはアイスの上部分をドラッグ＆ドロップして、コーンの上に順番に積み上げる。

指定された形が完成したらクリアとなり、派手な演出のあとで次の注文に進む。

---

## 3. 機能要件

### 3.1 アイス作成
- コーン付きのアイスクリームを作成できる
- アイスの上部分をドラッグ＆ドロップして、コーンの上に乗せる操作で組み立てる
- 指でのタッチ操作に対応する
- アイスは上に積み上げる形式とする
- 最大 10 段程度まで積めるようにする

### 3.2 見本表示
- 見本のアイスを枠内に表示する
- 見本は「お客さんの注文」らしい UI で表示する
- プレイヤーは見本を見ながら同じアイスを作成する

### 3.3 アイス種類
- アイスの種類は 10 種類前後を想定する
- 後から増やしたり減らしたりしやすい構造にする
- 色、形、味名などはデータ定義として分離する

### 3.4 注文生成
- お客さんの注文（見本）はランダムに生成する
- 最大 10 段までのランダムなアイス構成に対応する
- 連続プレイしても同じパターンになりにくいようにする

### 3.5 クリア演出
- 完成したら派手な演出を表示する
- クリア後はもう一度遊べるようにする
- 次の注文へ自然につながる構成にする

### 3.6 背景・見た目
- サーティワンアイスクリームのような店内をイメージした背景とする
- 子供向けにカラフルで楽しい見た目にする
- ボタンや操作対象は大きくする
- 誤操作しにくい UI にする

### 3.7 追加すると面白そうな機能案
以下は優先度順ではなく提案である。
- 連続正解でスコアやコンボが増える
- 完成したアイスに「ぴったり」「あと少し」などの評価を付ける
- 注文の待ち時間を表す簡単な演出を入れる
- アイスの完成後に客のリアクションを表示する
- 同じ種類のアイスを連続で置いたときにボーナス演出を出す
- クリア時に風船・星・紙吹雪などの演出を出す
- 難易度を段階的に上げる（最初は 3 段、慣れたら最大 10 段）
- 音のオンオフ切り替えを用意する

---

## 4. 非機能要件

### 4.1 対象・プラットフォーム
- 主対象は iPad の横画面
- 想定端末はホームボタン機を含む iPad の Safari
- 縦画面でも崩れないこと
- 子供が使う前提で、見た目と操作の分かりやすさを重視する

### 4.2 操作性
- 指での操作性を最優先する
- ボタンは大きくする
- 誤操作しにくい配置にする
- 1 回の操作で迷わないようにする
- 長押しメニュー、ページスクロール、ピンチ拡大などの誤操作対策を入れる

### 4.3 技術条件
- HTML / CSS / JavaScript のみで構成する
- 静的配信で GitHub Pages にそのまま置けること
- パスはすべて相対指定とする
- 外部依存を減らし、単体で動くようにする
- 後から機能追加しやすいよう、データとロジックを分離する

### 4.4 保守性
- アイスデータ、注文生成、描画、入力処理、演出を分ける
- 読みやすいコードを書く
- 後から種類追加・削除・調整がしやすい構造にする

### 4.5 公開場所
- GitHub リポジトリ `s3130029-blip/TSUMUGI-Games`
- 配置先は `<ゲーム名>/` サブフォルダ
- 他のゲームやリポジトリルートを壊さない

---

## 5. 実機確認の前提

### 5.1 実機確認の担当
- 作成者本人が確認する

### 5.2 実機確認のタイミング
- 骨組み段階で最初に確認する
- その後は大きな機能追加ごとに確認する
- 最後にまとめて確認するのではなく、開発途中から確認を始める

### 5.3 実機確認の場所・環境
- 手持ちの iPad
- Safari
- 必要に応じて GitHub Pages に仮公開した URL で確認する

### 5.4 iOS 固有のリスク
優先的に実機確認する対象は以下とする。
- タッチ操作の反応
- ドラッグ＆ドロップの追従性
- スクロールやズームの誤動作
- 長押しメニューの発生
- canvas を使う場合の描画品質やリサイズ挙動
- 画面回転時のレイアウト崩れ
- 音声を使う場合の再生制限や初回操作制限

---

## 6. 進め方

### 6.1 開発の進め方
1. まず動く最小構成を作る
2. 次にドラッグ＆ドロップでアイスを積めるようにする
3. 次に見本表示と注文生成を追加する
4. 次にクリア演出を追加する
5. 最後に調整、デザイン改善、追加機能を入れる

### 6.2 途中成果物
- `task.md` に作業チェックリストを作る
- 実装しながら `task.md` を更新する
- README または簡単な導入手順を付ける

### 6.3 迷ったときの方針
- 仕様が曖昧な箇所は、最もシンプルで子供に優しい挙動を採用する
- 採用理由は一言添える
- 複雑さより、まず遊べることを優先する

---

## 7. 受け入れ基準

最低限、以下を満たすこと。
- iPad Safari で遊べる
- コーン付きアイスをドラッグ操作で完成できる
- 見本を見ながら同じアイスを作れる
- 注文はランダム生成される
- クリア時に演出がある
- もう一度遊べる
- GitHub Pages でそのまま公開できる
- ルートや他ゲームを壊さず、指定サブフォルダ内で完結する
- 後からアイス種類や演出を追加しやすい

---

## 8. 実装メモ

- 最初は 3 段程度の簡単な注文から始めてもよい
- アイスの描画は、画像素材がなければ図形ベースで仮実装してよい
- 必要なら後から画像素材に置き換えられるようにする
- 子供向けなので、判定は厳しすぎない方針を優先する

# CLAUDE.md — アイスクリーム屋さんゲーム

このファイルは Claude Code がこのプロジェクトを実装するときの開発ルール・作業手順・禁止事項・優先順位をまとめたものです。
仕様の出典は [docs/requirements.md](docs/requirements.md)。仕様の詳細はそちらを正とし、このファイルは「どう作るか」の判断基準とします。

---

## 0. このプロジェクトを一言で

iPad（Safari）の子供が、注文見本どおりにコーンへアイスをドラッグ＆ドロップで積み上げて遊ぶ、**静的 Web アプリ**。
HTML / CSS / JavaScript のみで作り、**GitHub Pages のサブフォルダにそのまま置いて動く**ことがゴール。

---

## 1. 絶対に守る前提（技術的制約）

- **静的 Web アプリのみ。** HTML / CSS / JavaScript だけで構成する。
- **バックエンド・外部 API・サーバー処理は使わない。**
- **必須ビルドツールを使わない。** npm install しないと動かない構成にしない。`index.html` を開けば動くこと。
- **外部依存（CDN・ライブラリ）は原則使わない。** 単体（ファイルだけ）で動かす。どうしても必要なら理由をコメントで残し、最小限に。
- **パスはすべて相対指定。** `/assets/...` のような絶対パス禁止。`./assets/...` や `assets/...` を使う（サブフォルダ配信で壊れるため）。
- ターゲットブラウザは **iPad の Safari（ホームボタン機を含む）**。モダンすぎる API に依存しない。

---

## 2. 配置とスコープ（壊さないこと）

- このゲームは GitHub リポジトリ `s3130029-blip/TSUMUGI-Games` の **`icecream/` サブフォルダ内で完結**させる（このフォルダがそのままサブフォルダになる想定）。
- **リポジトリルートや他のゲームのファイルには絶対に触れない。** 変更・追加・削除はすべてこのフォルダ配下に限定する。
- GitHub Pages での想定 URL は `https://s3130029-blip.github.io/TSUMUGI-Games/icecream/`。**サブパス配信前提**でリンク・読み込みを書く（だから相対パス）。

## 3. iPad タッチ操作を最優先する

操作性はマウスより**指でのタッチを最優先**。実装の判断で迷ったら「指で気持ちよく操作できるか」を基準にする。

- **タッチ操作で組み立てられること**が必須。Pointer Events（`pointerdown` / `pointermove` / `pointerup`）で実装し、マウス・タッチを同じコードで扱う。`mousedown` 系単独に依存しない。
- ドラッグ＆ドロップは **HTML5 Drag and Drop API（`draggable`）を使わない**。iOS Safari で挙動が不安定なため、Pointer Events で自前実装する。
- **ボタン・アイス・コーンなど操作対象は大きく**する。指のタップ目標は最低 44×44pt 相当を確保。要素間に十分な余白を取り、誤タップを防ぐ。
- 1 回の操作で迷わない単純な動線にする。モーダルや多段メニューを増やさない。

---

## 4. 誤操作対策（重視・必ず入れる）

子供が触るので、ブラウザの標準ジェスチャによる事故を徹底的に抑える。**最初の最小構成の時点で入れておく。**

`index.html` の `<head>`：
```html
<meta name="viewport"
      content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover">
```

CSS（全体方針）：
```css
html, body {
  margin: 0;
  height: 100%;
  overflow: hidden;            /* ページスクロール抑制 */
  overscroll-behavior: none;   /* 引っ張り更新・バウンス抑制 */
  touch-action: none;          /* ピンチ拡大・ダブルタップズーム抑制 */
  -webkit-user-select: none;   /* テキスト選択抑制 */
  user-select: none;
  -webkit-touch-callout: none; /* 長押しメニュー抑制 */
  -webkit-tap-highlight-color: transparent; /* タップ時の青ハイライト消し */
}
```

JS（補助）：
- `contextmenu` イベントを `preventDefault()` して長押しメニューを抑制。
- ドラッグ中の `touchmove` / `pointermove` は必要に応じて `preventDefault()`（スクロール・ズーム巻き込み防止）。ただしハンドラは `{ passive: false }` で登録する。
- ダブルタップズーム対策は `touch-action: none` を基本とし、不足する端末向けに必要なら追加対応。

> これらは iOS Safari で挙動差が出やすい。**実機で必ず確認する**（セクション 8）。

## 5. モジュール構成（データとロジックを分離）

「アイスデータ」「注文生成」「描画」「入力処理」「演出」を**別ファイル・別責務**に分ける。後からアイス種類や演出を増減しやすくするため。

推奨構成（最小から始め、必要になったら分割する。最初から全部作らない）：
```
icecream/                 ← このフォルダ（リポジトリのサブフォルダ）
├─ index.html
├─ css/
│  └─ style.css
├─ js/
│  ├─ data/
│  │  └─ flavors.js       ← アイスの種類定義（色・形・味名）。データのみ、ロジックなし
│  ├─ order.js            ← 注文（見本）のランダム生成
│  ├─ render.js           ← 描画（DOM / canvas）
│  ├─ input.js            ← タッチ／ポインタ入力処理
│  ├─ effects.js          ← クリア演出など
│  └─ main.js             ← 全体の組み立て・状態管理
├─ assets/                ← 画像・音（後から追加。なくても動く）
├─ docs/
│  └─ requirements.md
├─ task.md                ← 作業チェックリスト（セクション 7）
├─ README.md              ← 簡単な導入手順
└─ CLAUDE.md
```

ルール：
- **アイスの種類（色・形・味名）はデータ定義として `js/data/` に分離**する。配列／オブジェクトの追加だけで種類を増減できる形にする。ロジックに色や名前をベタ書きしない。
- 注文生成・描画・入力・演出は互いの内部に踏み込まない。`main.js` が状態を持ち、各モジュールを呼び出す。
- 画像素材がなければ**図形ベース（CSS / canvas の塗り）で仮実装**してよい。後から画像へ差し替えられるよう、見た目はデータ／描画層に閉じ込める。

## 6. 作業手順（最小構成 → 段階的に追加）

**まず動く最小構成を作り、そこから少しずつ機能を足す。**一度に全部作らない。各ステップの完了時に動作確認できる状態を保つ。

1. **最小構成**：`index.html` を開くと画面が出る。誤操作対策（セクション 4）込み。コーン 1 つを表示するだけでよい。
2. **ドラッグ＆ドロップ**：アイスの上部分を指で掴んでコーンの上に積めるようにする（最大 10 段程度）。
3. **見本表示 + 注文生成**：注文見本を「お客さんの注文」らしく枠内に表示し、ランダム生成する。連続で同じ並びになりにくくする。
4. **クリア判定 + 演出**：見本どおりに完成したらクリア。派手な演出を出し、もう一度／次の注文へつなぐ。子供向けなので**判定は厳しすぎない**。
5. **調整・デザイン・追加機能**：背景（サーティワン風の店内イメージ）、カラフルな見た目、難易度段階、スコア／コンボ、音の ON/OFF などを必要に応じて。

各ステップで仕様に迷ったら、セクション 9 の方針で決める。

---

## 7. task.md の運用（必須）

- **まず `task.md` を作成**し、上記ステップを作業チェックリストに落とす。
- **実装しながら逐次更新**する（完了に `[x]`、新たに気づいたタスクは追記）。最後にまとめて書かない。
- 実装の途中経過・残課題が `task.md` を見れば分かる状態を保つ。
- あわせて `README.md`（または簡単な導入手順）を用意する。

---

## 8. 実機確認（早い段階から）

- **骨組み（最小構成）の段階で最初に実機確認**する。最後にまとめて確認しない。以降は大きな機能追加ごとに確認する。
- 環境：**手持ちの iPad / Safari**。必要に応じて GitHub Pages に仮公開した URL で確認する。
- 優先的に確認する iOS 固有リスク：
  - タッチ反応 / ドラッグ＆ドロップの追従性
  - スクロール・ズーム・引っ張り更新の誤動作
  - 長押しメニューの発生
  - canvas を使う場合の描画品質・リサイズ挙動
  - 画面回転時（横↔縦）のレイアウト崩れ
  - 音を使う場合の再生制限（初回ユーザー操作までは鳴らない等）
- 主対象は **iPad 横画面**だが、**縦画面でも崩れない**こと。

---

## 9. 迷ったときの判断方針

- 仕様が曖昧なら、**最もシンプルで子供に優しい挙動**を採用する。
- 採用理由を**一言だけ**コードコメントか task.md に添える。
- **複雑さより「まず遊べる」を優先**する。凝った実装より、確実に動いて分かりやすい挙動を選ぶ。
- 最初は 3 段程度の簡単な注文から始めてよい（慣れたら最大 10 段へ段階的に）。

---

## 10. 優先順位（上から順に守る）

1. **iPad Safari で実際に遊べること**（タッチ操作・誤操作対策）
2. **ルートや他ゲームを壊さない**こと（`icecream/` 内で完結、相対パス）
3. **受け入れ基準（セクション 12）を満たすこと**
4. **データとロジックの分離**（後から種類・演出を増やしやすい構造）
5. **見た目の作り込み・追加機能**（コンボ、評価、音 ON/OFF など）

迷ったら上位を優先。見た目や追加機能のために 1〜2 を犠牲にしない。

---

## 11. 禁止事項

- ❌ バックエンド／外部 API／サーバー処理を使う
- ❌ 必須ビルドツール（要 npm install／要バンドル）を前提にする
- ❌ 絶対パス（`/...`）でのリソース参照
- ❌ 外部 CDN・ライブラリへの安易な依存（原則ゼロ。必要時は理由を明記し最小限）
- ❌ HTML5 Drag and Drop API（`draggable`）への依存（iOS で不安定）
- ❌ リポジトリルートや他ゲームのファイルの変更・削除
- ❌ アイスの色・名前などのデータをロジックにベタ書きする
- ❌ 最小構成を飛ばして一気に作り込む／実機確認を最後まで後回しにする
- ❌ 子供向けなのに判定を厳しくしすぎる・操作を複雑にする
